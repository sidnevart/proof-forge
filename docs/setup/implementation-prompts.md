# Первые 10 промптов для реализации

1. С помощью `startup-positioning` определить ICP, основную боль и текущий внешний рабочий вариант имени `Checkpoint`, при этом сохранить `ProofForge` как внутреннее имя репозитория и операционного контура.
2. С помощью `brand-voice-system` и `visual-identity-director` расширить утверждённый identity package в переиспользуемые brand rules, которые остаются системными, proof-centric и явно не похожими на habit tracker или wellness.
3. С помощью `system-architect` предложить первую full-stack архитектуру для MVP, не размывая утверждённый accountability loop, границу buddy approval и proof-first framing.
4. С помощью `domain-modeler` определить goals, pacts, evidence, approvals, weekly recaps и progress status так, чтобы это отражало утверждённое позиционирование `Checkpoint`.
5. С помощью `api-contract-designer` зафиксировать начальные HTTP API и Telegram webhook contracts вокруг отправки proof, buddy review и изменений progress state.
6. С помощью `database-architect` спроектировать relational schema и подход к migrations для goals, proof artifacts, review outcomes и accountability relationships.
7. С помощью `go-project-bootstrapper` заскелетить Go backend вокруг утверждённой архитектуры и удержать реализацию в рамках brand constraints.
8. С помощью `web-app-bootstrapper` и `ui-kit-builder` заскелетить frontend shell так, чтобы первые UI primitives уже поддерживали surfaces уровня mission control, scoreboard и proof archive.
9. С помощью `telegram-bot-architect` и `telegram-webhook-builder` определить границу bot interaction так, чтобы она не превратилась в generic notification flow или habit-reminder flow.
10. С помощью `docker-compose-builder` и `ci-cd-builder` связать local и CI execution paths, а затем добавить brand-aware checks, которые не позволят будущей работе уйти от утверждённого identity package.
